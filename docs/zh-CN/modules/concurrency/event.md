# event

`event` 适合组织 **发布 / 订阅式事件流**。当系统里有明显的事件边界，希望让生产者与消费者解耦时，可以优先考虑这类模型。

## 适合什么场景

- 业务动作完成后要通知多个后续处理者。
- 想把同步主流程与异步扩展行为分开。
- 需要在模块之间建立更松耦合的协作关系。

## 使用建议

- 先定义清楚事件名称、负载结构和幂等语义，再考虑订阅数量。
- 事件驱动会提高扩展性，也会增加排障与链路理解成本。
- 如果只是并发执行几个固定任务，不一定需要上升到事件模型。
