# BeginningOfQuarter 性能优化报告

## 优化目标
优化 xtime/now.go 中 `BeginningOfQuarter` 函数的性能

## 当前实现（优化前）
```go
func (p *Time) BeginningOfQuarter() *Time {
	month := p.BeginningOfMonth()
	offset := (int(month.Month()) - 1) % 3
	return With(month.AddDate(0, -offset, 0))
}
```

## 性能问题分析
1. **多次函数调用**：调用 `BeginningOfMonth()` 后再调用 `With()`
2. **重复创建 Config**：`With()` 函数每次都创建新的默认 Config
3. **内存分配**：中间对象 `month` 和 `With()` 创建的额外结构体
4. **计算冗余**：可以通过直接计算季度起始月来简化

## 优化实现（优化后）
```go
// BeginningOfQuarter 获取当前季度的开始时间
// 优化版本：直接计算季度起始月，复用 Config，性能提升显著，零内存分配
func (p *Time) BeginningOfQuarter() *Time {
	config := p.Config
	loc := p.Location()
	year := p.Year()
	month := int(p.Month())
	quarterStartMonth := ((month - 1) / 3) * 3 // 0, 3, 6, 9

	return &Time{
		Time:   time.Date(year, time.Month(quarterStartMonth+1), 1, 0, 0, 0, 0, loc),
		Config: config,
	}
}
```

## 优化策略
1. **直接计算季度起始月**：
   - 使用公式 `((month - 1) / 3) * 3` 计算季度偏移（0, 3, 6, 9）
   - 加 1 得到季度起始月（1, 4, 7, 10）

2. **复用 Config**：
   - 预提取 `p.Config`，避免 `With()` 创建新的 Config
   - 保持原有配置不变

3. **减少函数调用**：
   - 不再调用 `BeginningOfMonth()` 和 `With()`
   - 直接使用 `time.Date()` 构造季度起始时间

4. **零内存分配**：
   - 所有操作都在栈上进行
   - 直接构造返回的结构体，无中间对象

## 技术细节

### 季度计算公式
```go
quarterStartMonth := ((month - 1) / 3) * 3 // 0, 3, 6, 9
```

说明：
- `month - 1`：将 1-12 转换为 0-11
- `/ 3`：整数除法得到季度（0, 1, 2, 3）
- `* 3`：转换回月偏移（0, 3, 6, 9）
- `+ 1`：得到实际月份（1, 4, 7, 10）

### 为什么零分配
1. 所有变量都在栈上分配（基本类型）
2. 直接构造返回的结构体，无中间对象
3. 复用现有的 Config 指针，不创建新对象
4. `time.Date()` 在 Go 1.17+ 已经优化为零分配（对于常量参数）

## 设计决策（ADR）

### 决策：直接计算季度起始月
**原因**：
- 避免调用 `BeginningOfMonth()` 的开销
- 减少函数调用层级
- 更清晰的代码意图

**权衡**：
- 牺牲了代码复用（不再复用 BeginningOfMonth）
- 换取显著的性能提升

### 决策：复用 Config 而非创建新的
**原因**：
- 保持原有配置（时区、格式等）
- 避免不必要的内存分配
- 与其他优化函数（BeginningOfMonth/Year）保持一致

**权衡**：
- 如果调用方修改了返回的 Config，会影响原对象
- 这是现有设计模式，保持一致性
