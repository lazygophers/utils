## 优化策略分析

### 原始实现瓶颈

```go
func mapGetWithSeparator(m map[string]any, key string, sep string) (any, error) {
    // 1. splitKey: 创建多个字符串分配
    parts := splitKey(key, sep)

    // 2. navigateToValue: 遍历调用，每次分配
    for _, part := range parts {
        val, err := navigateToValue(current, part)
        // ...
    }
}
```

**性能瓶颈**：
- `splitKey` 每个部分创建新字符串（堆分配）
- `navigateToValue` 额外函数调用开销
- 路径构建时的字符串拼接分配

### 优化版本策略

```go
func mapGetWithSeparatorOptimized(m map[string]any, key string, sep string) (any, error) {
    // 1. 快速路径：简单键直接返回
    if !strings.Contains(key, sep) && strings.IndexByte(key, '[') == -1 {
        return m[key], nil  // 零分配
    }

    // 2. 字节级解析：栈上数组存储位置
    var spans [16]span  // 栈分配，避免堆

    // 3. 内联导航：避免函数调用
    for i := 0; i < spanCount; i++ {
        part := key[spans[i].start:spans[i].end]  // 字节切片，零分配
        // 内联类型判断和访问
    }
}
```

**优化技巧**：
1. **快速路径**：简单键直接访问，避免解析
2. **字节级解析**：使用 span 结构存储位置，而非字符串切片
3. **内联导航**：展开 `navigateToValue` 逻辑，避免函数调用
4. **栈上分配**：固定大小数组存储中间结果
5. **延迟字符串创建**：仅在错误消息时才创建字符串
